# 技術選定

本アプリケーションは、以下の技術構成で開発を行う。

### フロントエンド

-   **フレームワーク**: React (Next.js)
    -   **理由**: Web開発で広く使われ、UIの再利用性が高く、複雑な画面構築に適している。Next.jsはReactをベースにしており、初期表示の高速化や将来的な機能拡張に柔軟に対応できます。
-   **言語**: TypeScript
    -   **理由**: コードの品質を高め、大規模開発での保守性を向上させます。
-   **UIライブラリ**: Chakra UI
    -   **理由**: モダンでアクセシビリティに配慮したUIを効率的に構築でき、画面レイアウトの再現に適しています。

### バックエンド

-   **言語**: Node.js
    -   **理由**: フロントエンドと同じJavaScript/TypeScriptで開発できるため、開発効率が良いです。非同期処理に強く、リアルタイム通信との相性も抜群です。
-   **フレームワーク**: Express
    -   **理由**: Node.jsの標準的なWebアプリケーションフレームワークで、シンプルで拡張性が高く、今回の要件に十分対応できます。

### リアルタイム通信

-   **ライブラリ**: Socket.IO
    -   **理由**: WebSocketを簡単に扱うためのライブラリで、ゲームのリアルタイムなイベント通知（プレイヤーの参加、入力の同期、投票結果など）に最適です。自動再接続機能もあり、安定した通信が期待できます。

### データベース

-   **初期実装**: バックエンドサーバーのメモリ上で管理
    -   **理由**: ゲームのルーム情報やプレイヤーの状態は、揮発性の高いデータなので、初期開発をシンプルかつ迅速に進めるため、まずはサーバーのメモリ上で管理します。（例: JavaScriptのMapオブジェクトなど）
-   **将来的な拡張**: Redis (高速インメモリDB) や MongoDB (お題の永続化)
    -   **理由**: 「お題の蓄積」などの永続化機能や、ゲームの状態をより堅牢に管理する必要が出た際に導入を検討します。

### デプロイ環境

-   **フロントエンド**: Vercel
    -   **理由**: Next.jsアプリケーションのデプロイに最適化されており、簡単に公開できます。
-   **バックエンド**: Render
    -   **理由**: Node.jsアプリケーション、特にWebSocketを使用するサーバーを簡単な手順で公開でき、無料プランもあります。
